# 📈 AISA (AI-driven ISA Portfolio Manager)

주소 : https://gixst.vercel.app/

> **낮은 변동성, 안정적인 수익률**을 위한 개인 맞춤형 포트폴리오 관리 서비스

## 📖 프로젝트 소개

재테크 열풍 속에서 많은 개인 투자자들이 주식 시장에 뛰어들고 있습니다. 하지만 대부분의 사람들이 어떤 종목과 상품에 얼마나 투자하는 것이 낮은 변동성을 가지면서 안정적인 수익률을 가질 수 있을지에 대한 고민이 많아 보입니다.
이러한 문제를 해결하기 위해 **개인 맞춤형 포트폴리오**를 설계하고 관리해주는 웹 서비스를 개발하고자 합니다.
단순한 종목 추천을 넘어, **변동성을 낮추고 안정적인 수익**을 낼 수 있는 자산 배분 전략을 제안합니다.

## 🚀 핵심 기능 계획

### 1. 📊 시장 데이터 분석 (Market Data)

- **거시 경제 지표**: M2 통화량, CPI 소비자 물가, 환율, 기준 금리, 채권 금리, 달러 환산 코스피 조회
- **AI 국내 지수 밸류에이션**: Gemini API를 활용한 국내 지수(코스피, 코스닥) 밸류에이션 조회
  - **밸류에이션 점수** : CAPE, Yield GAP을 고려하여 현재 시장이 고점/저점인가를 판단
  - **트렌드 점수** : 현물과 선물 순매수와 변동성 지수, 상승/히락 종목 수 추이를 고려하여 현재 시장이 상승/하락장인가를 판단
- **AI 국내/해외 기업 분석 리포트**: Gemini API를 활용한 특정 기업의 개요, 미래 성장 동력, 리스크 조회
- **AI 단기 모멘텀/장기 품질 리포트**: 기업의 주가가 현재 단기적으로 상승/하락 추세인지를 분석하는 리포트 / 기업의 펀더멘탈과 밸류에이션을 비교해서 투자 적합성을 분석하는 리포트

### 2. 💼 포트폴리오 관리 (Portfolio Management)

- **나만의 포트폴리오 구성**: 관심 종목으로 포트폴리오를 구성하고 관리
- **포트폴리오 배당 캘린더** : 포트폴리오 내의 종목들의 배당락일, 배당지급일, 배당금, 배당률을 달력 형태로 제공
- **포트폴리오 진단**:
  - **자산 상관관계 분석**: 포트폴리오 내 자산 간의 상관계수(Pearson, rolling) 분석 및 시각화 (히트맵)
  - **변동성 분석**: 롤링 beta/상관계수 분석으로 포트폴리오의 변동성 분석
- **백테스팅 (Backtesting)**: 구성한 포트폴리오의 과거 성과(수익률, MDD, Sharpe Ratio) 시뮬레이션

### 3. 💰 ISA 특화 기능 (ISA Specialization)

- **절세 효과 시뮬레이션**: 일반 계좌 대비 ISA 계좌 활용 시 예상 세금 절감액 계산

## 트러블 슈팅

### 1. 백테스트

문제 : 포트폴리오 내의 특정 종목의 상장일이 백테스트 시작일보다 이후일 경우 어떻게 처리해야 하는가?

해결 방안 : 해당 종목의 비율 만큼 현금 보유로 처리한 다음, 해당 종목의 상장일 이후 매수 처리를 해서 수익률 비교에 합산

</br>

### 2. Gemini API Key 라운드 로빈

문제 1 : 다수의 사용자들이 Gemini API를 사용해서 특정 기능을 호출할 때, 하나의 Key가 이미 사용 중일 경우 다른 사용자의 결과가 느려지거나 에러가 호출되는 문제(Rate Limit(429 Too Many Requests))가 발생

문제 2 : 무료 Gemini API Key의 경우 하루 20회 제한으로 적은 사용 횟수가 주어짐

추론 : 각 API Key는 동일한 Rate Limit 정책을 공유하므로, 논리적으로 동일한 성능을 가진 독립 노드로 간주함

해결 방안 : 서로 다른 계정에서 여러 프로젝트에서 생성된 Gemini API Key를 등록한 다음, 라운드 로빈 형식으로 수행

</br>

### 3. JPA N+1 문제 해결 (성능 최적화)

문제 : 포트폴리오 상세 조회 및 백테스트 진행 시, 포트폴리오 내 보유 종목(`PortStock`) 리스트를 불러와 개별 주식(`Stock`) 정보에 접근하는 과정에서 포트폴리오 내 주식 종목 개수(N)만큼 추가적인 DB SELECT 쿼리가 발생하는 성능 저하 현상 (N+1 문제) 발생.

추론 : JPA에서 `PortStock`과 `Stock`의 `@ManyToOne` 연관 관계를 지연 로딩(`FetchType.LAZY`)으로 설정함에 따라, `findByPortfolio` 등으로 리스트를 가져오는 쿼리 1번 이후 루프를 돌며 `.getStock().getStockCode()`에 접근할 때마다 프록시 객체를 초기화하기 위한 쿼리가 매번 실행됨. (종목이 10개면 총 11번의 쿼리 발생)

해결 방안 : `PortStockRepository`에 JPQL 상에서 연관된 엔티티를 한 번에 가져오는 **Fetch Join** 메서드(`SELECT ps FROM PortStock ps JOIN FETCH ps.stock ...`)를 명시적으로 선언하고 서비스 로직(`PortStockService`, `BacktestService`)의 조회 메서드를 교체하여, 단 하나의 쿼리로 연관 객체들까지 모두 로딩되도록 하여 데이터베이스 통신 횟수를 획기적으로 낮추고 응답 속도를 개선함.
